Overview — what is GTMDB?

For external integrations and client apps, use the REST API (GET /health, versioned routes under /v1/). Run the server with python -m gtmdb serve. Interactive OpenAPI docs live at /docs on the server. The sections below also describe the Python library used by the server and by in-process agents.

GtmDB is the full-stack data layer built for the era of GTM autonomous operations. While most agentic systems rely on scattered data and rigid tables without awareness of permissions, GTMDB —backed by multi-model storage—provides agents exactly what they need to make high-integrity, autonomous choices without exposing unnecessary information. Whether your agents are built on OpenAI Frontier, Claude Cowork, or LangGraph , GtmDB provides them with secure access to your business data, so they can run your business.

What GTMDB supports

  1. One system of record for GTM—and the agents that use it. Typed CRUD and traversals over the commercial graph—accounts through campaigns plus activities, comms, events, insights, and operational stats in one entity model—so one store can back MCP servers, LangChain-style tools, and bindings for common agent stacks (LangGraph, Vertex AI, OpenClaw, and similar) instead of every assistant re-wrapping a different partial API per system.
  2. Security, permissions, and access out of the box — for people and for agents. Every call runs under a Scope tied to declarative policies: tenant isolation, read/write rules, field-level masking, and redaction. Issue different tokens for a human rep, an internal agent, or an external partner so you control exactly what each principal sees and can change.
  3. Triggers and automation hooks. Design your stack so events in GTMDB drive downstream behavior: e.g. trigger an agent when a deal closes, when a campaign is sent, or when signals indicate learning or coaching moments—closing the loop between CRM state and autonomous workflows.
  4. Agents can ask anything — from facts, to analytical and statistical questions. The graph + indexed analytical store model supports exploratory and quantitative questions over your pipeline and GTM motion: strategic prompts (“why are we losing to competitors?” in context of your deals and accounts) and statistical views (funnels, attribution, stage distributions) on top of the same CRM-native store—not a separate BI silo.

Sales, marketing & customer success — one system of record

The entity model is deliberately cross-functional: the same graph holds what marketing, sales, and customer success (CS) care about, so GTMDB can serve as the ideal system of record for the whole revenue business —not three disconnected spreadsheets or siloed tools.

  • Marketing — Campaign, Lead, segments and lists, content and landing experiences, form and page-view events, attribution touchpoints, and links to Deal (INFLUENCED). Full-text search over accounts and programs; pipeline visibility stays tied to the same graph sales uses.
  • Sales — Account / Contact hierarchy, Deal, quotes, orders, products, contracts, comms (email, call, meeting), tasks, and partner accounts—plus traversals (360°, path-finding) for account planning and multi-threading.
  • Customer success & support — Tickets, SLAs, health scores, renewal context on the same Account graph as sales; timeline from emails, meetings, usage events, and notes so CS, sales, and marketing share one customer truth.
  • RevOps & leadership — Metric snapshots, forecasts, dashboards/reports, and AI or rule-driven insights materialized on top of the same store—no shadow warehouse required for operational GTM questions.

Because every department reads and writes through the same CRM-native API under Scope, you avoid duplicate “shadow” CRMs: one store, one permission model, one place agents and humans query when they need the full picture of a customer or campaign.

Under the hood, GTMDB stores CRM entities as nodes and relationships in Neo4j (labels such as Lead, Account, Deal, etc.). That choice enables graph traversals (360° views, timelines, attribution paths, search). Applications integrate through the GtmDB client; optional SQLAlchemy metadata defines an AccessToken table so you can persist issued tokens in your app database (CRM2 does this); the token row holds JSON policies that drive authorization.

What this library includes

  • GtmDB — async entrypoint: connect_gtmdb / close, typed sub-APIs (leads, accounts, contacts, deals, campaigns, emails, email_campaigns), relationships, low-level NodeData/EdgeData, and read helpers (search, pipeline, entity_360, explore_subgraph, …).
  • Typed entity layer — dataclasses (Lead, Account, …) and CRUD with filtering, updates, and convenience methods (e.g. contacts.assign_to_account, leads.link_campaign, campaigns.add_lead).
  • Scope — evaluates JSON policies for every operation: read/write on node labels and relationship types, optional per-record ids scoping, field-level masking (denylist / allowlist), and redaction when a row exists but must not be fully revealed.
  • AccessToken — SQLAlchemy model: tenant_id, owner_id, owner_type, policies (JSON string), redact_mode, label, is_active, timestamps.
  • create_token_from_presets + PRESETS — compose tokens from named policy bundles (full_access, read_all, write_all, no_raw_content) plus optional custom_presets and extra_policies.
  • Schema bootstrap — idempotent Neo4j constraints, indexes, and full-text index for entity search (run on connect()).
  • seed_sample_graph — demo data for tests and local development.
  • GtmdbSettings — Pydantic settings with GTMDB_ env prefix for Neo4j URI, credentials, key store, and admin key.

Every API method that touches data requires a Scope derived from a token. The same code path enforces tenant isolation (tenant_id on the token is applied to queries and writes) and authorization (policies). Admin keys grant full_access scope; agent keys are policy-scoped. Only admin scopes can manage API keys.

Entity API pattern CRUD

The unified GTM record spans companies (Account), people (Contact), prospects (Lead), opportunities (Deal), programs (Campaign, EmailCampaign, Email), and the wider types in GTM entity model—all linkable in one graph. The sections below use the shared create / get / list / update / delete pattern; examples pass a scope argument. Build a Scope from an access token as described under Permissions, scopes & access tokens (later on this page). Additional types evolve on the same contracts (schema, Scope, relationships)—including via low-level nodes and edges when you prototype ahead of generated clients.

Connection (API key required)

connect_gtmdb always requires a valid api_key. Both admins and agents use the same function. The library checks the admin key first (local match via GTMDB_ADMIN_KEY, no Postgres round-trip); if it doesn't match, the key is resolved from the Postgres key store. Returns (db, scope).

Empty or whitespace-only api_key raises ValueError. Invalid, expired, or revoked keys raise ValueError from resolution.

Agent connection
from gtmdb import connect_gtmdb

api_key = "your_api_key_here"  # provisioned by an admin
db, scope = await connect_gtmdb(api_key=api_key)

lead = await db.leads.create(
    scope, actor_id="my-agent", company_name="Acme", status="new"
)
await db.close()
Admin connection
import os
from gtmdb import connect_gtmdb

db, scope = await connect_gtmdb(api_key=os.environ["GTMDB_ADMIN_KEY"])
# Admin scope: full_access, owner_type="admin"
# Can manage API keys:
result = await db.api_keys.create(
    owner_id="sdr-agent", tenant_id=scope.tenant_id, preset_names=["write_all"],
)
print(result.raw_key)  # give this to the agent
await db.close()

Admin key: set GTMDB_ADMIN_KEY env var to a strong secret (e.g. python3 -c "import secrets; print(secrets.token_urlsafe(48))"). This key is checked locally and grants full_access scope with owner_type="admin". Only admin scopes can call db.api_keys.create/revoke/rotate/list; non-admin scopes receive PermissionError.

Agent key format: gtmdb_<key_id>_<secret> where <key_id> is a short URL-safe prefix and <secret> is the random token portion.

actor_id, audit edges, and tiered reasoning

actor_id (required, non-empty) on every create / update identifies the Actor performing the write. The API MERGEs that Actor, sets created_by_actor_id / updated_by_actor_id on the node, and creates (Actor)-[:CREATED_BY|:UPDATED_BY]->(entity) (direction: actor → entity). Optional keyword reasoning on create/update is stored on that audit relationship when non-empty.

Score domain reasoning (qualification notes on the Score node) is separate from relationship reasoning. Create scores only via db.leads.add_score(..., has_score_reasoning=..., reasoning=None, ...); ScoresAPI.create is not supported (raises TypeError).

Display name on nodes: Lead / Contact — composed person name with fallbacks to company / email; Email — subject or Email step {n}; Score — {score_type}:{total} from add_score; Deal, Account, Campaign — the name field you set.

Edge reasoning — required (non-empty) on: relationships.create; leads.link_campaign; contacts.assign_to_account; deals.assign_to_account, deals.add_contact; campaigns.add_lead; leads.add_score (has_score_reasoning for HAS_SCORE); and email_campaigns.create_with_artifacts when lead_ids is non-empty (sourced_from_reasoning).

Optional on: HAS_EMAIL in create_with_artifacts (has_email_reasoning); arbitrary low-level EdgeData; and purely structural edges you create yourself.

ACTOR = "my-agent-id"  # must exist or will be MERGEd as Actor

lead = await db.leads.create(
    scope,
    actor_id=ACTOR,
    reasoning="Imported from form",  # optional; CREATED_BY edge
    company_name="Acme",
    first_name="Jane",
    status="new",
)
row = await db.leads.get(scope, lead.id)
rows = await db.leads.list(scope, status="new", limit=20)
await db.leads.update(scope, lead.id, actor_id=ACTOR, status="qualified")
await db.leads.delete(scope, lead.id)

Account db.accounts

An Account is a company or organization you sell to: the canonical B2B “customer record.” It holds firmographic attributes (name, domain, industry, size, revenue). People (Contacts) and open opportunities (Deals) are linked to an account via relationships, not duplicated inside the account row.

Python type: from gtmdb import Account — dataclass with id, tenant_id, created_at, updated_at, plus name, domain, industry, employee_count, annual_revenue, website, type (e.g. prospect vs customer).

API surface: standard CRUD only — create, get, list, update, delete. There are no extra methods on AccountsAPI; use db.contacts.for_account / db.deals.for_account to query who works there and which deals belong to the account. 

from gtmdb import Account

# scope = Scope(token)  # see #permissions
acc = await db.accounts.create(
    scope,
    actor_id=ACTOR,
    name="Acme Industries",
    domain="acme.example",
    industry="Software",
    employee_count=120,
    type="customer",
)
# acc.id is assigned by GTMDB — store it for linking contacts/deals

loaded: Account | None = await db.accounts.get(scope, acc.id)
same_tenant = await db.accounts.list(scope, domain="acme.example", limit=10)

await db.accounts.update(scope, acc.id, actor_id=ACTOR, annual_revenue=5_000_000.0)
removed = await db.accounts.delete(scope, acc.id)  # bool

Contact db.contacts

A Contact is a person: name, email, title, and optional company string or department. In the graph model, a contact is linked to exactly one Account (typical pattern) via WORKS_AT using assign_to_account. Use for_account to list everyone at a company.

Python type: from gtmdb import Contact — name (auto-set on create/update), first_name, last_name, email, phone, title, company_name, department, linkedin_url, plus system fields on Entity.

Extra methods: assign_to_account(scope, contact_id, account_id, *, reasoning) — reasoning required, for_account(scope, account_id, limit=50). 

from gtmdb import Contact

person = await db.contacts.create(
    scope,
    actor_id=ACTOR,
    first_name="Jane",
    last_name="Doe",
    email="jane@acme.example",
    title="VP Sales",
    company_name="Acme Industries",
)
await db.contacts.assign_to_account(
    scope, person.id, acc.id, reasoning="HR directory match"
)

team: list[Contact] = await db.contacts.for_account(scope, acc.id, limit=100)
one = await db.contacts.get(scope, person.id)
await db.contacts.update(scope, person.id, actor_id=ACTOR, department="Revenue")
await db.contacts.delete(scope, person.id)

Lead db.leads

A Lead is a prospect not yet fully qualified as a contact/opportunity: often from inbound forms, ads, or research. Fields mirror a lightweight person+company record plus status (e.g. new, qualified) and optional source / score. To associate a lead with a company at the graph level, use db.relationships.create(..., "WORKS_AT", ...) if needed; contacts use the typed assign_to_account helper instead. Link a lead to a Campaign with link_campaign (SOURCED_FROM lead → campaign), or from the campaign side via REST POST /v1/campaigns/{id}/add-lead / campaigns.add_lead.

Python type: from gtmdb import Lead — company_name, first_name, last_name, email, phone, title, domain, status, source, score, linkedin_url.

Extra methods: link_campaign(..., *, reasoning) — SOURCED_FROM lead → campaign; add_score(..., *, actor_id, has_score_reasoning, reasoning=None, **score_fields) — creates Score + HAS_SCORE; scores_for — list scores for a lead. 

from gtmdb import Lead

prospect = await db.leads.create(
    scope,
    actor_id=ACTOR,
    company_name="Beta Corp",
    first_name="Sam",
    last_name="Lee",
    status="new",
    source="webinar",
    score=42.0,
)
await db.relationships.create(
    scope, prospect.id, "WORKS_AT", acc.id, reasoning="Clear firmographic match"
)
cmp = await db.campaigns.create(
    scope, actor_id=ACTOR, name="Webinar series", channel="webinar"
)
await db.leads.link_campaign(
    scope, prospect.id, cmp.id, reasoning="Registered for webinar"
)

queue = await db.leads.list(scope, status="new", limit=50, offset=0)
await db.leads.update(scope, prospect.id, actor_id=ACTOR, status="qualified")
await db.leads.delete(scope, prospect.id)

Deal db.deals

A Deal is a sales opportunity: amount, stage, probability, optional close date and owner. Attach it to an Account with assign_to_account (BELONGS_TO). Link Contacts involved via add_contact (HAS_CONTACT). Campaign influence edges (INFLUENCED) are not exposed via typed deal/campaign helpers; use db.relationships.create if your model requires them, or rely on GET /v1/entities/{id}/explore to read the surrounding graph.

Python type: from gtmdb import Deal — name, amount, stage, probability, close_date, description, owner_id. 

from gtmdb import Deal

opp = await db.deals.create(
    scope,
    actor_id=ACTOR,
    name="Acme — Enterprise renewal",
    amount=120_000,
    stage="negotiation",
    probability=0.7,
    close_date="2026-06-30",
    owner_id="rep-42",
)
await db.deals.assign_to_account(
    scope, opp.id, acc.id, reasoning="Renewal on existing customer"
)
await db.deals.add_contact(
    scope, opp.id, person.id, reasoning="Economic buyer on opportunity"
)

open_at_acme: list[Deal] = await db.deals.for_account(scope, acc.id)
await db.deals.update(scope, opp.id, actor_id=ACTOR, stage="closed_won")
await db.deals.delete(scope, opp.id)

Campaign db.campaigns

A Campaign is a marketing program (channel, budget, dates, status). Tie leads to a generic campaign with db.leads.link_campaign (SOURCED_FROM) or db.campaigns.add_lead (same edge, campaign-centric). For email programs with sequenced messages, use db.email_campaigns and create_with_artifacts instead.

Python type: from gtmdb import Campaign — name, status, channel, budget, start_date, end_date, description. 

from gtmdb import Campaign

cmp = await db.campaigns.create(
    scope,
    actor_id=ACTOR,
    name="Q1 Outbound",
    status="active",
    channel="email",
    budget=25_000,
)
await db.campaigns.add_lead(
    scope, cmp.id, lead.id, reasoning="Inbound from this program"
)

await db.campaigns.delete(scope, cmp.id)

EmailCampaign db.email_campaigns

An EmailCampaign is a marketing program specialized for email: it inherits all Campaign fields (name, status, channel forced to email on create_with_artifacts, budget, dates, description) plus from_name, from_email, and reply_to for the sending identity. Graph label EmailCampaign. Lives in gtmdb.api.communication_events.

Use create_with_artifacts to persist the campaign node, one Email node per artifact, HAS_EMAIL edges (email campaign → email), and optional SOURCED_FROM edges from Lead ids to the email campaign (same pattern as db.leads.link_campaign for generic Campaign nodes). 

from gtmdb import connect_gtmdb

db, scope = await connect_gtmdb(api_key="…")

out = await db.email_campaigns.create_with_artifacts(
    scope,
    actor_id=scope.owner_id,
    name="Q1 Nurture",
    from_name="Marketing",
    from_email="mkt@example.com",
    emails=[
        {"subject": "Hello", "body": "First touch"},
        {"subject": "Follow up", "body": "Second touch"},
    ],
    lead_ids=["uuid-of-existing-lead"],  # optional; if set, sourced_from_reasoning required
    sourced_from_reasoning="Enrolled from inbound",
)
ec = out["campaign"]
# out["email_ids"], out["linked_lead_count"]
MethodParametersDescription
create_with_artifactsscope, required actor_id, emails, optional lead_ids, sourced_from_reasoning (required if lead_ids), optional has_email_reasoning, plus campaign kwargsPersist EmailCampaign + Emails + edges (does not send mail)

Tokens need write on labels EmailCampaign and Email, and on relationship types HAS_EMAIL and SOURCED_FROM (e.g. full_access with resources: ["*"]).

Email db.emails

An Email node stores a single email artifact (sequence step, draft, or send record): subject, body, optional per-message from_name / from_email / reply_to, state (e.g. draft), sequence_number, send_at. Full CRUD via db.emails; batch attach to a program via db.email_campaigns.create_with_artifacts. For HubSpot-style marketing email fields (content widgets, subscription lists, A/B testing), see HubSpot Marketing Emails API as a reference for future extensions.

End-to-end: account → contact → deal

Typical order: create Account, then Contact and assign_to_account, then Deal, assign_to_account on the deal, and add_contact on the deal.

acc = await db.accounts.create(scope, actor_id=ACTOR, name="Acme", domain="acme.example")
c = await db.contacts.create(
    scope, actor_id=ACTOR, first_name="Jane", last_name="Doe", email="j@acme.example"
)
await db.contacts.assign_to_account(
    scope, c.id, acc.id, reasoning="Employed at account"
)
d = await db.deals.create(
    scope, actor_id=ACTOR, name="Acme expansion", amount=50_000, stage="qualification"
)
await db.deals.assign_to_account(scope, d.id, acc.id, reasoning="Opportunity on account")
await db.deals.add_contact(scope, d.id, c.id, reasoning="Champion on deal")

GTM entity model

GTMDB is intended to subsume the entire revenue operating model: not only accounts and deals, but everything that happens around them—conversations, campaigns, product signals, support, and the metrics and narratives you derive from that activity. The types below share the same design language as the CRUD sections: naming, relationships, and typical links to Account, Contact, Deal, and related nodes. Typed db.* entrypoints roll out over time; until one exists for a label, you can model it with low-level nodes and edges under Scope.

Quote

Formal commercial offer: line items, pricing, validity, status (draft/sent/accepted/lost). Links to Account, Contact, and typically one or more Deals.

Order

Won business captured as a transaction: totals, billing/shipping refs, fulfillment state. Links from Quote or Deal, to Account and Line item rows.

Line item

Atomic row on a quote or order: Product reference, quantity, unit price, discounts. Enables revenue, margin, and product-mix reporting without denormalizing headers.

Product

SKU or sellable item: name, family, price book hooks, entitlements. Connects quotes, orders, and product usage events for adoption and expansion signals.

Subscription

Recurring commercial relationship: term, seat count, MRR/ARR rollups, renewal dates. Bridges Account, Contract, and billing Invoices.

Contract

Legal or commercial agreement: effective dates, auto-renew, governing terms. Anchors Subscription and renewal workflows for sales and CS.

Invoice

Money owed or collected: amounts, due dates, payment status, sync ids from finance systems. Links to Order / Subscription and Account.

Partner account

Channel, reseller, or tech partner org: tiers, MDF, co-sell flags. Relates to Account (customer) and Deal for partner-sourced pipeline.

Quota

Target envelope for a rep, team, or product line: period, amount or ARR goal, attainment rollups. Anchors Forecast and pipeline views without spreadsheet exports.

Territory

Routing and ownership rules over accounts or geos: named coverage, assignment policy, overlap handling. Links to Team rosters and Account for fair distribution and reporting.

Team

Sales or CS pod: members, manager hierarchy, cost center, named coverage of Territory. Used for forecasts, quotas, and “who owns this account” rollups.

Activity

Generic timeline row (subject, time, owner, type) that can normalize or wrap emails, calls, meetings, tasks—so timeline-style APIs have a single polymorphic backbone.

Email

Message metadata and body refs: direction, thread id, participants mapped to Contact, engagement signals. Feeds sequences, compliance, and conversation intelligence.

Call

Voice or video touchpoint: duration, disposition, recording/transcript refs, attendees. Links to Account, Deal, and Contact.

Meeting

Scheduled or completed calendar event: time, location/link, internal vs external attendees. Core to sales motion and CS QBRs.

Task

Owned follow-up: due date, priority, queue, completion. Relates to any anchor (Deal, Ticket, Account).

Note

Free-text capture with author and visibility; often child of Account, Contact, or Deal for handoffs and audit.

Chat message

In-app, Slack, or support-chat utterances: channel, sender, thread. Complements Email for full conversational context.

Domain event

Immutable something happened record: type, payload, occurred_at, correlation id, source system. Backplane for automation, analytics, and replay—distinct from CRM “state” rows.

Webinar / session

Hosted or sponsored session: registration list, attendance, recording. Links to Campaign, Lead, and Contact for demand gen attribution.

Form submission

Inbound capture event: form id, UTM, field payload, consent flags. Spawns or updates Lead and ties to Page view sequences.

Page view

Anonymous or identified web/app surface hit: path, session, referrer. Supports digital body language and scoring before a known Contact exists.

Product usage event

In-product telemetry: feature, volume, account/user id, timestamp. Drives adoption metrics, PQLs, and CS health without a separate product-analytics silo.

Segment

Named audience definition (rule or list membership) over people or accounts. Powers campaigns, sales prioritization, and agent “who to reach” tools.

Marketing list

Static or synced recipient set for a send or experiment; links to Campaign and Segment.

Content asset

Collateral with metadata: ebook, deck, demo clip, landing URL. Tracks engagement via events and ties to Campaign and Deal enablement.

Ticket

Support or success case: priority, status, SLA clock, category. Links to Account, Contact, and optionally Deal / Subscription.

Health score

Time-stamped snapshot of account or subscription health: dimensions (usage, support load, NPS), composite score, explainability refs. Distinct from ad-hoc Insight text.

Insight

Machine- or human-authored finding: summary, severity, evidence pointers (entities, events, metrics), recommended action. Suitable for agent surfacing and review queues.

Metric snapshot

Materialized stat at a grain (e.g. daily pipeline by stage, campaign ROI, seat utilization): measure name, dimensions, value, period. Feeds dashboards without re-scanning raw events every time.

Report / dashboard

Saved analytical view: definition (query spec or widget layout), owner, refresh policy. Optional export to BI; first-class in GTMDB for in-product revops.

Forecast

Relationship properties & reasoning

EdgeData and db.create_edge accept optional reasoning; when non-empty it is stored on the relationship. Typed APIs enforce non-empty reasoning where business judgment applies — see actor_id, audit edges, and tiered reasoning.

  • EdgeData(..., reasoning="…") or db.create_edge(scope, EdgeData(..., reasoning="…")) — optional unless your caller contract says otherwise.
  • relationships.create(..., reasoning="…", **props) — reasoning is required and non-empty.
  • Entity create / update: optional reasoning on the CREATED_BY / UPDATED_BY audit edge (from actor_id).
  • Typed CRM helpers listed under tiered rules — required reasoning (or has_score_reasoning / sourced_from_reasoning where noted).
  • leads.add_score — required has_score_reasoning for HAS_SCORE; optional reasoning for CREATED_BY on the score node; domain qualification text is the Score field reasoning in **kwargs.
  • email_campaigns.create_with_artifacts — optional has_email_reasoning; if lead_ids non-empty, sourced_from_reasoning is required.

relationships.list returns each relationship’s properties dict (including reasoning when present). get_neighbors and each item in entity_360’s connected list include edge_properties: the property map of the traversing relationship (again including reasoning).

API Key Management

GtmDB has two types of API keys: an admin key (set via GTMDB_ADMIN_KEY env var, checked locally) and agent keys (stored in PostgreSQL, policy-scoped). Both are passed to connect_gtmdb(api_key=...). Only admin scopes can create, revoke, rotate, or list agent keys — agents receive PermissionError if they attempt key management.

Types: ApiKeyResult and ApiKeyInfo

ApiKeyInfo On key listing / read key_id, owner_id, owner_type, label, is_active, expires_at, created_at, last_used_at

db.api_keys — API Key Manager

Access via db.api_keys after connecting. All methods are async. Admin methods (create, revoke, rotate, list_keys) require owner_type="admin" scope; non-admin callers get PermissionError. resolve has no guard (used internally at connect time).

result = await db_admin.api_keys.create(

)
await db_admin.close()
print(result.raw_key)  # give this to the agent (shown once)

# Step 2: Agent connects with the provisioned key

await db.close()

Security model

  • Hashing: Raw API keys are hashed with SHA-256 before storage. The raw_key is shown only once at creation; the system never reveals it again.
  • Storage: Persisted in PostgreSQL table gtmdb_api_keys with columns: key_id (unhashed prefix for lookup), key_hash (SHA-256), tenant_id, owner_id, owner_type, policies (JSON), is_active, expires_at, created_at, last_used_at.
  • Expiration: Optional expires_at timestamp. Resolution fails if the key is past expiration.
  • Revocation: Set is_active=false to block a key without deleting it. Resolution fails for revoked keys.
  • Policies: Each key is bound to a set of JSON policies ("full_access", "read_all", "write_all", or custom). These are enforced when the key is resolved into a Scope.
  • Tenant isolation: Every key resolution checks that tenant_id matches the token, preventing cross-tenant leaks.
  • No in-memory cache: Each resolve() performs a direct PostgreSQL lookup on the indexed key_id.

Permissions, scopes & access tokens

Authorization is declarative: you attach a JSON list of policy objects to an AccessToken, then wrap that token in a Scope. The graph adapter consults the scope on every read and write (nodes, edges, traversals, raw Cypher executed through the client).

Policy object shape

Each policy is a dict. Typical fields:

KeyMeaning
effect"allow" or "deny"
actionsList including "read" and/or "write"
resourcesList of resource strings (see below)
conditionsOptional dict; supported: ids — list of record ids for instance-level rules

Resource strings

  • "*" — matches any resource (whole types).
  • "Lead", "Deal", … — node label (case-insensitive match in evaluator). Used for entity CRUD and reads.
  • "WORKS_AT", "INFLUENCED", … — relationship type for edge create/delete and write checks.
  • "deal.amount" — field-level resource: <label>.<property> (lowercase label in practice). Used with read for allowlist/denylist masking.

Evaluation rules (summary)

  • Read / write — Policies whose actions include the requested action and whose resources match the requested resource (or field prefix) are collected. deny on the whole type (* or exact label) blocks access. allow must be present for access to be granted.
  • conditions.ids — If present, the policy only applies when the checked instance includes {"id": "<value>"} and that id is in the list. Useful for “this token may only read these deal ids”.
  • Field masking (mask_fields) — On read, if there is a type-level read allow, deny policies on deal.secret_field strip those keys from returned properties. If there is no type-level allow but only field-level allow policies, the result is allowlist mode: only allowed fields remain.
  • Redaction (redact_mode on token) — If the principal may not read a node, the adapter either returns a stub node with {"_redacted": true} (hint) or omits it entirely (hide), depending on AccessToken.redact_mode.

Built-in presets (PRESETS)

NameEffect
full_accessAllow read+write on *
read_allAllow read on *
write_allAllow write on *
no_raw_contentDeny read on sensitive field resources (e.g. email body, transcripts)

Merge presets and add your own via create_token_from_presets(..., custom_presets={...}) and extra_policies=[...].

AccessToken row

Tokens are usually persisted in your app database (SQLAlchemy). The AccessToken model carries tenant_id, owner_id, owner_type, a JSON string of policies, label, is_active, optional expires_at (for API key–derived tokens), and timestamps. Load the row and pass it to Scope(...) per request or agent session; the graph adapter never bypasses what that row allows. Redaction is always set to "hint" mode (redacted nodes show {"_redacted": true}).

Real-world permission examples

Examples align with issuing per-agent AccessToken rows in apps such as CRM2. Policy approach is shorthand for JSON policies (effect, actions, resources, optional conditions.ids for row-level scope). Use label.field resources (e.g. deal.amount, email.body) for masking. The last column is what the principal sees after mask_fields and redaction.

import uuid

from gtmdb import ActorSpec

tenant = uuid.UUID("00000000-0000-4000-8000-000000000001")
token = create_token_from_presets(
    tenant,
    owner_id="demand-gen-agent",
    owner_type="agent",
    preset_names=["full_access"],
    label="token:demand_gen",
    redact_mode="hint",
)
scope = Scope(token)

db = GtmDB(GtmdbSettings())
await db.connect()
await db.actors.create(
    scope,
    [ActorSpec(id="demand-gen-agent", kind="ai", display_name="Demand gen")],
)
lead = await db.leads.create(
    scope, actor_id="demand-gen-agent", company_name="Acme", status="new"
)
await db.close()

For production, persist AccessToken rows via SQLAlchemy, load the row, and construct Scope(AccessToken(...)) per request or session. GTMDB does not cache scopes; you can issue one scope per agent or user as your app requires.
